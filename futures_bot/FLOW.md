# 실거래 봇 동작 흐름 (main.py)

`python main.py` 실행 시 실제로 어떤 주기로, 어떤 종목을, 어떤 조건으로 매매하고,
언제 텔레그램 알림이 오는지 정리한 문서. 파라미터는 [config/settings.py](config/settings.py) 기본값 기준이며,
그 파일만 고치면 아래 수치도 함께 바뀐다.

## 1. 대상 종목 & 체크 주기

| 항목 | 값 | 위치 |
|---|---|---|
| 대상 종목 | BTC/USDT:USDT, ETH/USDT:USDT (둘 다 독립적으로 처리) | `CONFIG.symbols` |
| 레버리지 | 각 5배, ISOLATED 마진 | `SymbolConfig` |
| 캔들 타임프레임 | 15분봉 | `StrategyConfig.timeframe` |
| 폴링 주기 | 30초마다 전체 종목을 순회 | `AppConfig.poll_interval_sec` |
| 모드 | One-way Mode 전제 (Hedge Mode 계정이면 주문 거부됨) | — |

**중요**: 30초마다 도는 건 "포지션이 방금 청산됐는지 감시"하기 위해서다. 신규 진입/반전 같은
매매 판단 자체는 **15분봉이 새로 확정될 때만** 내린다 (같은 캔들이 진행 중이면 스킵).
즉 최악의 경우 신호가 생겨도 최대 30초 이내에는 인지하지만, 그 신호 자체는 15분마다만 새로 계산된다.

## 2. 전체 루프 (심볼별로 매 폴링마다 반복)

```
매 30초, 심볼(BTC, ETH)마다:

  1) [상시] 보유 중이던 포지션이 거래소에서 사라졌는지 확인
       └ 사라졌다면 → 손절(STOP_MARKET)/익절(TAKE_PROFIT_MARKET) 중 뭐가 체결됐는지 조회
                     → 남은 반대쪽 예약주문 취소 → 실현손익 계산 → [청산] 알림

  2) 최근 15분봉 300개 수집
       └ 아직 같은 캔들이 진행 중이면 여기서 종료 (이번 폴링에서 더 할 일 없음)

  3) [15분봉 새로 확정될 때만] 시그널 계산 (EMA/ATR/RSI, trend_breakout.py)

  4) 현재 포지션 상태와 시그널을 비교해 행동 결정
       ├ 시그널 없음/같은 방향 유지     → 아무 것도 안 함
       ├ 포지션 없음 + 시그널 있음      → 신규 진입 시도
       └ 반대 방향 시그널              → 기존 청산 + 반대 방향 재진입 (반전)

  5) 신규 진입/반전 시 리스크 체크
       ├ 일일 손실 한도 / 연속손실 쿨다운 걸림 → 진입 안 하고 [진입 차단] 알림
       ├ 손절가가 청산가에 너무 가까움 / 최소 주문금액 미달 → [진입 스킵] 알림
       └ 통과 → 시장가 진입 + 손절/익절 예약주문 동시 실행 → [진입] 또는 [반전] 알림

  6) 이 과정에서 예외 발생 시 → 해당 심볼만 건너뛰고 [에러] 알림, 다음 폴링에서 계속
```

## 3. 매매 전략 — `trend_breakout` (strategies/trend_breakout.py)

3가지 조건을 **모두** 만족해야 신호가 발생한다 (AND 결합, 단일 지표보다 휩쏘 감소가 목적).

| 필터 | 조건 (LONG) | 조건 (SHORT) | 파라미터 |
|---|---|---|---|
| 추세 | EMA20 > EMA60 (정배열) | EMA20 < EMA60 (역배열) | `ema_fast=20`, `ema_slow=60` |
| 변동성 돌파 | 종가 > 볼린저밴드 상단 | 종가 < 볼린저밴드 하단 | `bb_period=20`, `bb_num_std=2.0` |
| 모멘텀 확인 | RSI > 52 이고 상승 중 | RSI < 48 이고 하락 중 | `rsi_period=14`, 임계값 52/48 |

- RSI는 역추세 매매용이 아니라 항상 "추세 방향과 같은 쪽으로 가고 있는지" 확인 용도로만 쓴다.
- 판단은 항상 **확정된 캔들**(진행 중인 마지막 봉 제외)만 사용해서 미확정 데이터로 진입하지 않는다.
- 돌파선(볼린저밴드) 계산도 "직전 확정봉까지"의 값만 써서 전방참조를 막는다.
- ATR(14)은 더 이상 돌파 신호에는 쓰이지 않지만, 손절폭 계산(risk 레이어, ATR×1.5)에는 계속 사용된다.

## 4. 리스크 관리 (risk/)

| 항목 | 값 |
|---|---|
| 1회 거래당 손실 한도 | 계좌자본의 0.5% (`risk_per_trade_pct`) — 포지션 수량은 이 한도를 손절폭으로 역산 |
| 손절폭 | 진입가 ∓ ATR×1.5 (`atr_stop_multiplier`) |
| 익절폭 | 손절폭 × 1.8 (`reward_risk_ratio`) — R:R = 1:1.8 |
| 청산가 안전마진 | 손절가가 추정 청산가보다 최소 20% 더 안쪽에 있어야 진입 허용 (`liquidation_safety_margin_pct`) |
| 일일 손실 한도 | 계좌자본의 -3% 도달 시 당일 신규 진입 전면 중단 (`max_daily_loss_pct`) |
| 연속 손절 쿨다운 | 연속 4회 손절 시 120분간 신규 진입 중단 (`max_consecutive_losses`, `cooldown_minutes`) |

이 한도들은 `logs/account_guard_state.json`에 날짜 단위로 저장되며, 봇을 재시작해도 당일 상태는 유지된다.

## 5. 주문 실행 (execution/order_manager.py)

진입 시 아래 3개 주문을 순서대로 넣는다 — 프로세스가 죽어도 거래소 서버에 걸린 예약주문으로
손절/익절이 보장되도록 하기 위함이다.

1. 시장가 진입 (`buy`=LONG, `sell`=SHORT)
2. `STOP_MARKET` 손절 예약 (reduceOnly)
3. `TAKE_PROFIT_MARKET` 익절 예약 (reduceOnly)

> 이 두 예약주문은 OCO로 묶여있지 않다. 하나가 체결되면 봇이 다음 폴링(최대 30초 이내)에
> 포지션이 사라진 걸 감지해서 남은 반대쪽 주문을 직접 취소한다.

## 6. 텔레그램 알림 — 언제, 무엇을

`.env`에 `TELEGRAM_BOT_TOKEN`/`TELEGRAM_CHAT_ID`를 넣으면 아래 6가지 상황마다 자동 전송된다
(`logs/trade_logger.py`). 알림이 실패해도 거래 로직에는 영향 없음.

| 상황 | 트리거 | 라벨 |
|---|---|---|
| 신규 진입 성공 | 리스크 체크 통과 후 주문 3건 실행 완료 | 🟢 [진입] |
| 청산 (자동 SL/TP 체결) | 폴링 중 보유 포지션이 사라진 걸 감지 | 🔴 [청산] (사유: 익절/손절) |
| 반전 (반대 시그널) | 기존 청산 + 반대 방향 재진입까지 성공 | 🔁 [반전] (청산+진입 통합 메시지) |
| 반전인데 재진입 실패/차단 | 청산은 했으나 재진입은 리스크 체크에 걸림 | 🔴 [청산] (사유에 재진입 실패/차단 명시) |
| 진입 차단 | 일일 손실 한도 또는 쿨다운 중 신호 발생 | ⏸️ [진입 차단] |
| 진입 스킵 | 청산가 근접 / 최소 주문금액 미달 등 | ⏭️ [진입 스킵] |
| 에러 | 폴링 중 심볼 처리 예외 발생 | ⚠️ [에러] |

각 메시지에는 가격/수량/손절가/익절가/실현손익(USDT, %)/그날 누적 손익%/사유가 포함된다.
예시:

```
🟢 [진입] BTC/USDT LONG
가격: 67,250
수량: 0.012
손절: 66,480
익절: 68,636
리스크: 계좌 0.5%
오늘 손익: +1.24%
사유: EMA상승 + 볼린저밴드 상단돌파 + RSI상승
```

## 7. 알려진 한계

- **스냅샷은 메모리에만 존재**: 포지션을 보유한 채로 봇 프로세스를 재시작하면, 그 포지션의
  진입가/손절가/익절가 정보가 사라져서 이후 SL/TP 자동 체결을 감지하지 못한다(알림 누락).
  재시작이 잦다면 파일 영속화 추가가 필요.
- **청산가 추정치는 근사치**: `risk/stop_loss.py`의 청산가 계산은 단순화된 공식이라, 실제
  Binance notional bracket 기준과 다를 수 있다. 안전마진(20%)이 최종 방어선은 아니다.
- 이 문서는 **실거래 루프(main.py)** 기준이며, 백테스트(`run_backtest.py`)는 완전히 별개
  경로로 동작하고 텔레그램 알림을 보내지 않는다.
