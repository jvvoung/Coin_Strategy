# Binance USDT-M Futures 양방향 통합 전략 설계 문서 (strategy_design.md)

> 본 문서는 기존 보유 중인 65개 전략 코드(주식/코인 현물/업비트/바이낸스 등 혼재)를 분석하여,
> 그 안의 공통 로직과 재사용 가능한 구조를 추출한 뒤, **Binance USDT-M Futures 기준 Long/Short 양방향 자동매매 통합 전략**을
> 새로 설계하기 위한 개발 방향 문서다. 실제 매매/백테스트 코드는 이 문서 승인 이후 별도 단계에서 작성한다.

---

## 0. 전제 및 스코프

- 대상 거래소: **Binance USDT-M Futures** (현물 아님)
- 포지션 방향: **Long / Short 양방향** (단방향 전략 금지)
- 기존 자료: 프로젝트 폴더 내 `0.` ~ `65.` 번호가 매겨진 65개 하위 폴더. 각 폴더는 국내주식/업비트/바이낸스 현물·선물 등 서로 다른 시장을 다루는 실전 봇 코드(`*_Bot.py`, `*_Test.py`, `*_bot.py` 등)를 포함하고 있음을 사전 확인함.
- 본 문서 자체는 "설계 방향"만 다루며, 코드는 포함하지 않는다. 실제 구현은 이후 단계(`config/`, `data/`, `indicators/`, `strategies/`, `risk/`, `execution/`, `backtest/`)에서 순차적으로 진행한다.

## 1. 전체 프로젝트 목적

### 1.1 왜 65개를 분석하는가
- 65개 전략은 시장(국내주식/미국주식/업비트/바이낸스), 상품(현물/선물), 방향성(단방향/양방향), 스타일(추세추종/평균회귀/변동성돌파/분할매매/자산배분 등)이 전부 다르다.
- 목적은 **"어떤 전략이 제일 좋은가"를 고르는 것이 아니라**, 서로 다른 65개 구현체에 반복적으로 등장하는 **엔지니어링 패턴과 리스크 처리 로직**을 뽑아내는 것이다.
  - 예: 캔들 수집 → 지표 계산 → 시그널 → 포지션 체크 → 주문 → 로그, 라는 파이프라인은 65개 중 대다수가 형태만 다르게 반복 구현하고 있을 가능성이 높음.
  - 이 반복 구조를 하나의 표준 아키텍처로 뽑아내면, 이후 어떤 신규 전략을 추가하더라도 "시그널 함수 하나만 교체"하는 수준으로 확장 가능해진다.

### 1.2 무엇을 하지 않는가
- 65개 전략을 그대로 복사/이식하지 않는다.
- 특정 1개 전략을 "우승 전략"으로 뽑아 그대로 쓰지 않는다.
- 백테스트 성과(연복리 수익률 등)만으로 채택 여부를 판단하지 않는다. (기존 폴더명에 "연복리 200%", "연복리 27%" 등의 표현이 많은데, 이런 수치는 국내주식/업비트 현물 기준이며 **선물 레버리지·청산·펀딩비 환경에서는 그대로 재현되지 않음**을 전제로 한다.)

### 1.3 최종 목표
- Binance Futures 계정 하나로 운용 가능한 **단일 통합 전략 모듈**을 설계한다.
- 이 모듈은:
  - Long/Short 신호를 동시에 평가할 수 있고,
  - 여러 하위 전략(추세추종/평균회귀/돌파 등)을 스위치처럼 켜고 끌 수 있고,
  - 리스크 관리 레이어가 전략 로직과 분리되어 있어, 어떤 하위 전략을 쓰든 동일한 손절/포지션사이즈 규칙이 적용되고,
  - 실거래 코드와 백테스트 코드가 시그널 함수를 100% 공유한다(로직 이원화로 인한 실전-백테스트 괴리 방지).

## 2. 65개 전략 코드 분석 방식

### 2.1 전략별 분석 항목 (체크리스트)
각 전략 코드를 아래 항목으로 분해하여 표준화된 카드 형태로 정리한다.

| 항목 | 세부 내용 |
|---|---|
| 기본 정보 | 파일명, 대상 시장(주식/업비트/바이낸스 현물·선물), 코드 성격(`_Bot`=실거래, `_Test`=백테스트) |
| 사용 지표 | MA/EMA, RSI, MACD, Bollinger Band, ATR, 거래량, 모멘텀 스코어 등 |
| 진입 조건 | Long 조건식, Short 조건식(있다면), 필터 조건(추세 필터, 변동성 필터, 거래대금 필터 등) |
| 청산 조건 | 익절 조건, 손절 조건, 반대 시그널 청산, 시간 기반 청산(타임아웃 청산) |
| 손절/익절 방식 | 고정 %, ATR 배수, 트레일링 스탑, 분할 익절 여부 |
| 포지션 관리 | 단일 포지션 vs 분할매수(무한매수/매직스플릿), 피라미딩 여부, 동시 보유 종목 수 |
| 시간봉 | 사용 타임프레임(1m/5m/15m/1h/1d), 멀티타임프레임 여부 |
| 리스크 관리 | 레버리지 사용 여부, 1회 손실 한도, 일일 손실 제한, 연속 손실 대응 |
| 필터 조건 | 거래대금/거래량 필터, 시장 상황 필터(횡보장 회피 등), 뉴스/이벤트 회피(할트 전략류) |
| 코드 구조 | API 호출 방식, 주문 재시도/예외처리 유무, 로그 기록 방식, 스케줄링 방식(폴링 주기) |
| 실전 적용성 | 아래 2.3 기준으로 평가 |

### 2.2 전략 간 장단점 비교 방법
- 65개를 위 체크리스트로 표로 만든 뒤, 다음 세 가지 축으로 교차 비교한다.
  1. **로직 축**: 진입/청산 로직이 몇 개의 원형(추세추종형, 평균회귀형, 돌파형 등)으로 수렴하는지 클러스터링.
  2. **리스크 축**: 손절 유무, 레버리지 유무, 포지션 사이즈 계산 유무를 기준으로 "선물 이식 난이도" 채점.
  3. **엔지니어링 축**: 예외처리/재시도/로그 수준을 기준으로 "코드 재사용 가능성" 채점.
- 비교 결과는 하나의 마스터 표(65행)로 정리하고, 이 표가 3번(전략 분류) 및 10번(산출물) 항목의 기초 자료가 된다.

### 2.3 실전성(Binance Futures 적용 가능성) 평가 기준
"좋다/나쁘다"가 아니라, 아래 8개 항목에 대해 **적용 가능(A) / 조건부 적용 가능(B) / 적용 불가·재설계 필요(C)** 로 채점한다.

1. **수수료 구조 반영 여부**: 진입+청산 왕복 수수료(taker 0.04~0.05% 등)를 감안해도 순수익이 남는 빈도의 전략인가. 스캘핑성 전략일수록 C로 갈 확률이 높음.
2. **슬리피지 내성**: 시장가 진입 전제 전략이 저유동성 심볼/저유동성 시간대에서도 체결 품질을 유지하는가.
3. **급변동 대응**: 롱숏 전환 시 반대 포지션 청산 지연으로 인한 순간 이중 리스크 노출 가능성이 있는가(예: 급락 중 숏 전환 신호가 늦게 반영되는 구조).
4. **청산가(liquidation) 리스크**: 전략이 가정하는 손절폭이 사용 레버리지 대비 청산가보다 항상 안쪽에 있는지(즉 손절이 청산보다 먼저 걸리는 구조인지).
5. **레버리지 리스크**: 코드에 레버리지가 하드코딩되어 있는지, 심볼별 최대 레버리지/명목가 한도(Binance Notional Bracket)를 고려하는지.
6. **API 오류 내성**: 재시도 로직, 타임아웃 처리, rate limit(429/418) 대응이 있는지.
7. **주문 미체결 처리**: 지정가 미체결 시 재주문/취소 로직이 있는지, 반쯤 체결된 상태(partial fill)를 다루는지.
8. **포지션 중복 진입 방지**: 동일 심볼에 대해 이미 포지션이 있는 상태에서 같은 방향 신호가 다시 뜰 때 중복 진입을 막는 가드가 있는지.

이 8개 채점 결과를 종합해 "선물 이식 시 그대로 재사용 가능한 로직"과 "반드시 재설계해야 하는 로직"을 구분한다. 이 구분이 4~7번 설계의 근거가 된다.

### 2.4 중복 로직 분류 방법
- 진입 조건식을 정규화(지표 조합 형태로 추상화: 예 "단기MA > 장기MA" + "RSI < 임계값")하여 동일 패턴군으로 묶는다.
- 65개 중 다수가 이동평균 크로스/모멘텀 스코어/변동성 돌파의 변형일 가능성이 높으므로, 이 3개 원형을 우선 추출하고 나머지를 부속 분류로 정리한다.
- 완전히 동일한 로직의 파일명 변형(`_V2`, `_FixV`, `_Test`)은 "버전 히스토리"로 취급해 최신 버전만 대표로 채택한다.

## 3. 전략 분류 기준

65개를 아래 7개 카테고리로 1차 분류하고, 각 카테고리 내에서 "선물 이식 가능성 등급(A/B/C)"을 함께 표기한다.

1. **추세추종 전략**: MA/EMA 크로스, 모멘텀 스코어, 듀얼모멘텀 계열. Long/Short 전환이 자연스러워 양방향 이식이 쉬움.
2. **평균회귀 전략**: RSI 과매수/과매도, 볼린저밴드 회귀. 급변동 구간에서 역추세 진입이므로 청산가 리스크 평가가 특히 중요.
3. **돌파 전략(변동성 돌파 포함)**: 전일 레인지 돌파, 채널 돌파류(래리 윌리엄스 변동성 돌파 계열 포함). 가짜돌파(휩쏘) 리스크와 슬리피지 평가 필요.
4. **변동성 기반 전략**: ATR 기반 포지션사이즈/손절, 변동성 필터로 진입 자체를 스킵하는 전략.
5. **거래량 기반 전략**: 거래대금 급증 필터, 거래량 스파이크 기반 진입/회피(할트 전략류 포함).
6. **멀티 지표 조합 전략**: MA+RSI+MACD+거래량 등을 동시에 쓰는 필터형 전략. 자산배분/무한매수 계열(분할매수, 매직스플릿) 다수가 여기 포함되나, 이들은 원래 "레버리지 없는 분할 진입" 전제이므로 선물 이식 시 레버리지/청산 재설계가 필수.
7. **머신러닝/통계 기반 전략**: 별도 존재 시 분류(회귀/분류 모델, 통계적 임계값 자동 산출 등). 존재 여부는 65개 전수 분석 후 확정.

> 참고: 무한매수/매직스플릿/자산배분 계열(폴더명 기준 다수 확인됨: 예 "이평무한매수봇", "매직스플릿", "구간분할전략")은 현물·레버리지 없음을 전제로 한 분할매수 전략이므로, 그대로 선물에 옮기면 "물타기 + 레버리지"가 되어 청산 리스크가 기하급수적으로 커진다. 이 계열은 **로직만 참고**하고 실제 포지션 사이징은 리스크 모듈(7번)에서 전면 재설계한다.

## 4. 최종 전략 코드 설계 방향

### 4.1 전체 파이프라인
```
[Binance Futures WS/REST 캔들 수집]
        ↓
[지표 계산 레이어]
        ↓
[전략 시그널 레이어] → Long 시그널 / Short 시그널 / No-signal
        ↓
[포지션 상태 조회] (현재 포지션 없음 / Long 보유 / Short 보유)
        ↓
[시그널 vs 포지션 상태 매칭 로직] → 신규진입 / 유지 / 청산 / 반대전환
        ↓
[리스크 관리 레이어] (포지션사이즈, 손절가, 익절가 계산 + 사전 승인)
        ↓
[주문 실행 레이어] (시장가/지정가, 재시도, 체결확인)
        ↓
[로그/기록 레이어] (거래로그, 에러로그, 성과로그)
```

### 4.2 설계 원칙
- **시그널 함수는 실거래/백테스트가 100% 동일 코드를 사용**한다. 데이터 소스(실시간 API vs 과거 DataFrame)만 다르게 주입한다.
- 전략 로직(어떤 지표로 진입할지)과 리스크 로직(얼마나 진입할지, 언제 손절할지)을 반드시 분리한다. 65개 전략 다수가 이 둘을 한 함수에 섞어놓은 것으로 예상되며, 이것이 재사용성을 떨어뜨리는 주된 원인이므로 통합 설계에서는 명확히 분리한다.
- 모든 하위 전략(추세추종/평균회귀/돌파 등)은 동일한 시그널 인터페이스(예: `-1, 0, 1` 또는 `SHORT/NONE/LONG` enum)를 반환하도록 표준화하여, 여러 하위 전략을 조합(투표/가중합)하거나 심볼별로 다르게 배정할 수 있게 한다.

## 5. 코드 아키텍처 설계

```
project_root/
├── config/
│   ├── api_keys.py / .env       # Binance API Key/Secret (환경변수, 코드에 하드코딩 금지)
│   ├── symbols.yaml             # 거래 심볼 목록, 심볼별 레버리지/최소주문단위
│   ├── timeframes.yaml          # 전략별 사용 타임프레임
│   └── risk_config.yaml         # 1회 손실한도, 일일 손실한도, 연속손실 제한 등
│
├── data/
│   ├── binance_client.py        # REST/WS 클라이언트 래퍼 (재시도, rate-limit 처리 포함)
│   ├── candle_fetcher.py        # 과거/실시간 OHLCV 수집, 캐싱
│   └── realtime_feed.py         # WS 실시간 가격/포지션/주문 상태 스트림 관리
│
├── indicators/
│   ├── moving_average.py        # SMA/EMA/WMA
```

## 6. Long/Short 양방향 전략 설계

### 6.1 Long/Short 조건 분리 방식
- `strategies/base_strategy.py`에서 각 전략은 `generate_signal(df) -> {"long": bool, "short": bool, "score": float}` 형태의 표준 반환값을 갖는다.
- Long 조건과 Short 조건은 **완전히 대칭일 필요가 없다.** (예: 추세추종은 대칭 가능하지만, 돌파 전략은 상방 돌파와 하방 돌파의 필터 강도를 다르게 둘 수 있음). 다만 반환 인터페이스는 항상 동일해야 조합 레이어(`signal_combiner.py`)에서 처리 가능하다.

### 6.2 동시 중복 진입 방지
- 같은 심볼에 대해 한 틱(캔들 확정 시점)에서 시그널 평가 → 주문 실행까지를 **단일 잠금(lock)** 으로 묶는다. 주문 실행 중에는 동일 심볼에 대한 재평가를 큐에 넣지 않고 스킵한다.
- 주문 실행 전 반드시 `position_manager.py`를 통해 "현재 이 심볼에 진행 중인 미체결 주문이 있는가"를 재확인(주문 전 최종 조회)한 뒤 진입한다. 이는 65개 코드 검토 시 누락 여부를 중점적으로 확인할 항목이다(2.3의 8번 기준).

### 6.3 기존 포지션이 있을 때 반대 시그널 처리(반대 포지션 전환)
표준 상태 전이표:

| 현재 포지션 | 새 시그널 | 처리 |
|---|---|---|
| 없음 | Long | 신규 Long 진입 |
| 없음 | Short | 신규 Short 진입 |
| Long 보유 | Long (동일) | 유지 (피라미딩은 리스크 모듈에서 별도 허용 시에만) |
| Long 보유 | Short | ① 기존 Long 청산 → ② 청산 체결 확인 → ③ Short 신규 진입 (동시 처리 금지, 반드시 순차 + 체결 확인) |
| Short 보유 | Long | 위와 대칭 |
| 보유 중 | No-signal | 유지, 손절/익절 조건만 감시 |

- 반대 전환 시 "청산과 신규진입을 한 번의 시장가로 합쳐서 보내는" 방식(리버스 주문)은 Binance Futures API에서 가능하지만, **부분체결/슬리피지 급증 구간에서 의도치 않은 순간 노출**이 생길 수 있으므로 기본값은 "청산 완료 확인 후 신규 진입"으로 설계하고, 지연 허용범위(예: 최대 2초)를 설정값으로 둔다.

### 6.4 Hedge Mode vs One-way Mode 비교

| 항목 | One-way Mode | Hedge Mode |
|---|---|---|
| 동일 심볼 Long+Short 동시 보유 | 불가 | 가능 |
| 반대 시그널 처리 | 자동으로 기존 포지션과 상계(net) | 별도 포지션으로 분리 관리, 명시적 청산 필요 |
| 코드 복잡도 | 낮음 (포지션 하나만 추적) | 높음 (positionSide=LONG/SHORT 별도 추적) |
| 오류 시 리스크 | 상대적으로 예측 쉬움 | 헤지 포지션 관리 실수 시 양쪽 수수료+펀딩비 이중 부담 위험 |
| 본 프로젝트 적합성 | **적합 (기본 채택 권장)** | 조건부: 별도의 "동시 양방향 헤지 전략"(48번 폴더의 헤지봇 계열처럼)을 명시적으로 추가할 때만 |

- **권장안**: 기본 통합 전략은 **One-way Mode**로 설계한다. 이유: 통합 전략의 목표가 "시그널에 따라 Long ↔ Short 전환"이지 "Long/Short 동시 보유로 델타 중립을 유지하는 헤지"가 아니기 때문. 순수 헤지 전략(예: 스팟-퍼프 헤지)은 별도 모듈로 분리하고 통합 전략과 계좌/리스크 한도를 공유하지 않도록 설계한다.

### 6.5 Binance Futures 양방향 구현 시 주의점
- One-way Mode에서는 반대 방향 주문이 기존 포지션을 자동 상계(reduce)하므로, "청산 의도"와 "신규 반대진입 의도"를 코드에서 수량 계산으로 명확히 구분해야 한다(상계 수량 vs 신규 수량을 합산해서 한 번에 보낼지, 나눠 보낼지 결정).
- 심볼별 `positionSide`, `marginType`(isolated/cross), `leverage`는 주문 전에 매번 확인하지 말고 **전략 시작 시 1회 설정 후 캐싱**, 단 API가 거부하는 경우(이미 포지션 있을 때 레버리지 변경 불가 등) 예외 처리 필요.
- 최소 주문 수량/명목가(`minNotional`), 가격/수량 소수점 자리수(`tickSize`, `stepSize`)를 심볼별로 반드시 조회 후 라운딩해야 주문 거부를 방지한다.
- 펀딩비 정산 시각(8시간 간격) 근처에서 신규 진입 시 예상치 못한 비용이 발생할 수 있으므로, 펀딩비 필터(정산 임박 시 진입 보류 옵션)를 설정 가능하게 둔다.

## 7. 리스크 관리 설계

| 항목 | 설계 방향 |
|---|---|
| 1회 거래당 손실 한도 | 계좌 자본 대비 고정 비율(예: 0.5~1%)로 설정. 손절가와 진입가의 % 차이를 역산해 포지션 사이즈를 결정(고정 손실액 기반 사이징). |
| 레버리지 설정 | 심볼별 상한을 `config/symbols.yaml`에 명시. 전략이 요구하는 변동성 대비 과도한 레버리지 요청 시 자동 하향 조정(캡). |
| 손절가 계산 | 기본은 ATR 배수(예: entry - 1.5×ATR) 또는 최근 스윙 고/저점 기준. 고정 % 손절은 백테스트 비교용 옵션으로 병행 제공. |
| 익절가 계산 | 손익비(R:R) 기준 자동 산출(예: 손절폭의 1.5~2배) + 부분 익절(분할 청산) 옵션. |
| ATR 기반 동적 손절 | 변동성 확대 구간에서는 손절폭을 넓히고(휩쏘 방지), 변동성 축소 구간에서는 좁혀서(자본 보호) 시장 상황에 적응. |
| 일일 최대 손실 제한 | 하루 누적 손실이 계좌의 N%(예: 3%) 도달 시 당일 신규 진입 중단(`account_guard.py`). |
| 연속 손실 제한 | 연속 N회(예: 4회) 손절 발생 시 일정 시간(cooldown) 신규 진입 중단, 이후 포지션 사이즈 축소 재개(단계적 정상화). |
| 포지션 사이즈 계산 | `size = (계좌자본 × 1회손실한도%) / (진입가-손절가 거리) / 레버리지 반영`. 계산 결과가 `minNotional` 미만이면 진입 스킵(무리한 최소단위 진입 방지). |
| 청산가 안전마진 | 계산된 손절가가 해당 레버리지에서의 청산가보다 항상 일정 마진(예: 청산가 대비 20% 이상 여유) 안쪽에 있는지 사전 검증하는 가드 추가. |

## 8. 백테스트 설계

### 8.1 원칙
- 실거래 이전에 반드시 백테스트 → (가능하면) 페이퍼 트레이딩 → 소액 실거래 순서를 강제하는 구조로 설계(9번과 연결).

### 8.2 비용 반영
- **수수료**: Maker/Taker 요율을 심볼/주문타입별로 반영(시장가=Taker 고정 가정, 지정가 체결 시 Maker 요율 옵션).
- **슬리피지**: 고정 슬리피지(bp) 모델을 기본으로 하되, 변동성(ATR) 연동 슬리피지 모델을 옵션으로 제공(급변동 구간 과소평가 방지).
- **펀딩비**: 8시간 간격 펀딩비율을 과거 데이터로 반영, 포지션 보유 기간에 비례해 비용/수익 가감.
- **레버리지**: 명목 포지션 규모와 증거금을 구분 계산하고, 청산가 도달 시뮬레이션(강제청산) 로직을 포함해 "손절 전에 청산되는" 케이스를 별도로 카운트.

### 8.3 성과 지표
- 승률(Win Rate), 평균 손익비(Avg R:R), Profit Factor, 최대낙폭(MDD), Sharpe Ratio, Sortino Ratio, 거래당 평균 수수료/펀딩비 비중, 청산 발생 횟수(있다면 치명적 결함으로 별도 표기).

### 8.4 과최적화 방지
- 파라미터 그리드서치 결과를 "성과 상위 1개"가 아니라 **성과 상위 구간의 파라미터 안정성(민감도 분석)** 으로 평가한다. 파라미터를 ±10~20% 흔들었을 때 성과가 급격히 붕괴하면 과최적화로 간주하고 채택하지 않는다.
- 지표/조건 개수를 필요 이상으로 늘리지 않는다(멀티 지표 조합 전략 채택 시 각 지표가 실제로 성과 개선에 기여하는지 개별 검증).

### 8.5 Train/Test 분리 및 Walk-forward
- 전체 과거 데이터를 Train(파라미터 탐색) / Test(검증, 미공개) 기간으로 분리하고, Test 구간 성과만 최종 판단 근거로 삼는다.
- 이후 Walk-forward 방식(예: 6개월 학습 → 1개월 검증을 롤링)으로 시장 국면 변화에 대한 강건성을 재검증한다. `backtest/walk_forward.py`에서 담당.

## 9. 실거래 전 검증 단계

1. **코드 분석**: 65개 전략 개별 분석(2번 항목 체크리스트 적용) 및 마스터 비교표 완성.
2. **전략 통합**: 원형 3~4개(추세추종/평균회귀/돌파/변동성)로 수렴한 로직을 `strategies/`에 표준 인터페이스로 구현.
3. **백테스트**: 8번 설계에 따라 수수료/슬리피지/펀딩비/청산 반영 시뮬레이션 수행.
4. **파라미터 검증**: 민감도 분석 + Walk-forward로 과최적화 여부 확인.
5. **모의투자(Testnet) 또는 소액 실거래**: Binance Futures Testnet에서 실제 API 흐름(주문/체결/포지션 조회) 전체 리허설 → 통과 후 실계좌 최소 금액(예: 계좌 자본의 1~2%)으로 실거래 개시.
6. **로그 검증**: 거래로그/에러로그가 실제 체결 내역(Binance 거래 내역)과 100% 일치하는지 대사(reconciliation).
7. **예외처리 확인**: 강제 API 오류 상황(네트워크 끊김, rate limit, 주문 거부) 인위 재현 테스트.
8. **실거래 확대 순서**: 소액 검증 기간(예: 최소 2~4주 또는 최소 거래 횟수 N회 이상) 성과가 백테스트 기대치 범위 내일 때만 단계적으로 자본 배분 확대. 확대는 한 번에 배수 증액하지 않고 단계적(예: 2배씩)으로 진행.

## 10. 최종 산출물 계획

1. **65개 전략 분석 요약표** — 2번 체크리스트 기준 65행 × 항목 열, 실전성 등급(A/B/C) 포함.
2. **전략별 핵심 로직 정리 문서** — 카드 형태(전략명/분류/핵심 조건식/재사용 판정) 65개.
3. **공통 패턴 기반 최종 전략 설계서** — 본 문서(`strategy_design.md`)의 확장/확정판.
4. **Binance Futures 자동매매 코드 구조** — 5번 아키텍처에 따른 실제 모듈 스켈레톤.
